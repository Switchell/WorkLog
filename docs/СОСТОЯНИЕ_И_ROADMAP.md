# WorkLog (AXIS ERP) — аудит, пробелы и план роста

Документ обновляется по мере развития проекта. **Секретов здесь нет:** пароли только в `.env`.

---

## 1. Краткая оценка

| Критерий | Оценка | Комментарий |
|----------|--------|-------------|
| Функциональность для SMB / внутреннего учёта | **Высокая** | Учёт часов, ставки, импорт, PDF, Telegram, Grafana, клиентский портал |
| Инфраструктура (Docker, nginx, healthcheck) | **Хорошо** | Compose с зависимостями, тома, прямой доступ к портам для отладки |
| Качество кода и сопровождаемость | **Среднее** | Крупный монолит `app.py`, DDL «на старте» вместо миграций |
| Тесты и CI | **Низкое** | Несколько unit-тестов, нет пайплайна |
| Документация | **Смешанная** | `КОНТЕКСТ_ПРОЕКТА`, `DEPLOY`, `CONTRIBUTING` сильные; часть README устарела |
| Безопасность для продакшена | **Требует доработки** | Сессии Streamlit, дефолтный admin в коде, секреты в старых копиях README |

**Итог:** проект **сильнее типичного «скрипта на Streamlit»** (стек, интеграции, Grafana), но до уровня **продукт / SaaS / compliance** не хватает инженерной дисциплины: автоматизированных проверок, миграций БД, разбиения UI и явной модели безопасности.

## 2. Что уже есть (инвентаризация)

### Приложение (`src/`)

- **Админка** `app.py` — авторизация (bcrypt), роли admin/user, CRUD-логика учёта, импорт Excel, финансы, графики Plotly, PDF, бэкапы через UI, отчёты в Telegram.
- **Клиентский портал** `app_client.py` — отдельное Streamlit-приложение с паролями из env (`worklog_client_auth.py`).
- **Модули:** `worklog_finance.py`, `worklog_import_db.py`, `worklog_pdf.py`, `worklog_telegram*.py`, `worklog_charts.py`, `backup.py` (CLI), логирование ошибок.

### Данные

- **PostgreSQL**, пул SQLAlchemy с `pool_pre_ping` и настраиваемым пулом (`worklog_db.py`).
- Таблицы `work_logs`, `employee_rates`; упрощённые миграции через `CREATE TABLE` / `DO $$` в `init_connection()`.

### Инфраструктура

- **Docker Compose:** `db`, `admin`, `client`, `nginx`, `grafana`, `certbot`; проброс портов, healthcheck (после правок — корректный health клиента).
- **Grafana:** provisioning datasource + дашборд в репозитории, том `grafana_data`.
- **Nginx** — единая точка входа, разделение `/` и `/client/`.

### Тесты

- `tests/test_worklog_finance.py` — расчёты и rollup.
- `tests/test_worklog_import.py` — валидация импорта.
- `tests/test_worklog_config.py` — URL БД и пароли клиентского портала.

### Документация (сильные места)

- `docs/КОНТЕКСТ_ПРОЕКТА.md` — нюансы Docker, БД, Grafana, Telegram.
- `docs/DEPLOY.md` — чеклист развёртывания.
- `CONTRIBUTING.md` — запуск, тесты, модули.

## 3. Что нужно доработать (пробелы и риски)

### Критично для продакшена

1. **Секреты в документации** — не хранить реальные пароли в README; использовать только `.env.example` и ссылки на переменные.
2. **Учётная запись admin** — bcrypt-хеш в `worklog_auth.py` задаёт дефолтного пользователя; для прод: обязательная смена пароля, либо только `ADDITIONAL_USERS` / вынесение в секрет-хранилище.
3. **База на хосте** — порт `5432` наружу: в прод ограничить firewall / не публиковать Postgres.
4. **Отсутствие CI** — нет автоматического `pytest` при PR/пуше.

### Архитектура и сопровождение

5. **Монолитный `app.py`** — сложно ревьюить и тестировать; нет чёткого разделения «сервис / UI».
6. **Миграции БД** — DDL в приложении вместо Alembic (или аналога): риск рассинхрона сред и дрейфа схемы.
7. **Покрытие тестами** — нет интеграционных тестов с БД, нет тестов импорта/бэкапа (частично отмечено в `TODO_NEXT_STEPS.md`).

### Операции

8. **Pin образов** — в `TODO_NEXT_STEPS`: зафиксировать тег `certbot` (как у остальных сервисов).
9. **Мониторинг** — Grafana для аналитики данных есть; метрик приложения/хоста (Prometheus, алерты) — нет.
10. **Бэкапы** — есть создание/восстановление; нет описанного сценария off-site и проверки восстановления по расписанию.

### Документация

11. **`docs/README.md`** — устаревшие имена каталога (`work-tracker`), команды без `--env-file`, ранее — явный пароль; нужно синхронизировать с `DEPLOY.md` и `.env.example`.

## 4. План выхода на более высокую планку

Ниже — по фазам: от «безопасный минимум» к «продуктовый уровень».

### Фаза A — гигиена и доверие (1–2 недели)

- Убрать секреты из всех markdown; единый блок «учётные данные только из `.env`».
- Включить **CI** (GitHub Actions / аналог): `pytest`, при желании `ruff`/`black` на `src/`.
- Зафиксировать версии образов в compose (`certbot` и любые `latest`).
- Короткий **runbook**: «первый запуск», «смена пароля БД», «сброс Grafana admin».

### Фаза B — инженерная зрелость (2–6 недель)

- Вынести тяжёлую логику из `app.py` в пакет модулей (вкладки = отдельные функции/файлы).
- Ввести **Alembic** (или скрипт миграций): одна команда `upgrade` при деплое.
- Расширить тесты: импорт Excel (фикстуры), `calculate_finances` граничные случаи, smoke с **testcontainers** PostgreSQL (по желанию).
- Добавить `@st.fragment` там, где перерисовка всего экрана мешает (`TODO_NEXT_STEPS.md`).

### Фаза C — прод и масштаб (по необходимости)

- **Reverse proxy:** rate limiting, заголовки безопасности, отдельный доступ к админке по VPN или IP allowlist.
- **Наблюдаемость:** логи в единый сборщик, health-эндпоинты, алерты по сбоям контейнеров.
- **Продукт:** аудит действий (кто менял данные), экспорт API только после отдельной модели угроз.
